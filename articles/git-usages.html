Git常用操作

Posted on 2016-12-11(星期日) 20:24 in Tools

对于开源项目的开发来说，熟练使用git的应该算是必备技能。本篇博客记录了平时工作中较常用的git操作。

git log

  • 显示所有历史记录
# git log
  • 显示最近N条历史记录
# git log -N
  • 显示详细历史记录，包括代码改动内容
# git log -p
  • 只显示一行，哈希值和提交说明
# git log --pretty=oneline
  • 显示提交历史记录的简单图形
# glog
# git log --pretty=format:"%H %s" --graph
  • 自定义格式显示历史记录
# git log --pretty=format:" "

常用的格式占位符写法及其代表的意义如下：

选项   说明
%H  提交对象（commit）的完整哈希字串
%h  提交对象的简短哈希字串
%T  树对象（tree）的完整哈希字串
%t  树对象的简短哈希字串
%P  父对象（parent）的完整哈希字串
%p  父对象的简短哈希字串
%an 作者（author）的名字
%ae 作者的电子邮件地址
%ad 作者修订日期（可以用 -date= 选项定制格式）
%ar 作者修订日期，按多久以前的方式显示
%cn 提交者(committer)的名字
%ce 提交者的电子邮件地址
%cd 提交日期
%cr 提交日期，按多久以前的方式显示
%s  提交说明

git rebase

git rebase操作简单理解就是“换基”。一般只操作还未push到远程仓库的commit，一旦push到了远程仓库，那么不允许再修改commit，不然会给其他开发带来很多麻烦。

       Assume the following history exists and the current branch is "topic":

                     A---B---C topic
                    /
               D---E---F---G master


       From this point, the result of either of the following commands:

           git rebase master
           git rebase master topic

       would be:

                             A'--B'--C' topic
                            /
               D---E---F---G master

如上图所示，topic和master有共同的基E，即topic分支是从master分支的E点切出来的新分支。当topic分支有了自己的私有开发，但master分支仍然在更新时，若操作者需要拉取master分支更新的内容，这时就需要“换基”。也就是说，将topic的HEAD变成master分支最新的G，将自己的私有代码放到master分支的尾巴上。

git rebase还有一些高级且很重要的操作。

commit合并

当需要在一个分支上将两个或者以上的commits合并时，我们可以通过git rebase -i来完成这件事。

  • -i 是 --interactive 的简写，在使用 git rebase -i 时，我们要在后面再添加一个参数，这个参数应该是 最新的一个想保留的 Commit。

如下是一个简单的范例，将e46b345和44e910a两次commits合并成一个。

➜  nova git:(master) git log --pretty=format:"%H| %s"
e46b345324a8411e8b9b1a085f3df67e48799c48 | Fix novncproxy for python3
44e910a693b424a2f75fbca76f97eb6d8ed94fc7 | Tests: improve assertJsonEqual diagnostic message
e8b7214946fd124e568495cd7cd78ae3e0033d07 | Raise correct error instead of class exist in Placement API
...

根据上面的解释，那么-i后面的commit就是e8b7214。（若想将44e910a和e8b7214合并的话，则-i后面的commit应该是e46b345）

git rebase -i e8b7214

进入如下编辑界面：

pick 44e910a Tests: improve assertJsonEqual diagnostic message                                                                                                                                                                              
pick e46b345 Fix novncproxy for python3

# Rebase e8b7214..e46b345 onto e8b7214

#  f, fixup = like "squash", but discard this commit's log message
#  x, exec = run command (the rest of the line) using shell

更换commits顺序

在本地基于某个分支做开发时，由于总会不断的完善功能，很容易出现一些小功能之间穿插提交代码，造成commits比较混乱。git rebase -i依然可以很好地解决这个问题。

以下范例是需要将dev1-2和dev2-1调整顺序。

➜  tests git:(master) git log --pretty=format:"%h | %s"
12be243 | dev2-2
da0a1d4 | test
22a5a50 | dev1-2
14bbd74 | dev2-1
98e752f | dev1-1

➜  tests git:(master) git rebase -i 98e752f

pick 14bbd74 dev2-1
pick 22a5a50 dev1-2
pick da0a1d4 test
pick 12be243 dev2-2

将以上内容显示的顺序与git log查看的顺序完全相反。只需将dev1-2和dev2-1调换顺序即可。调换顺序后保存退出，查看历史记录：

➜  tests git:(master) git log --pretty=format:"%h | %s"
ff2c926 | dev2-2
f894e78 | test
8482018 | dev2-1
6743e79 | dev1-2
98e752f | dev1-1

若想将test这次的commit删除，同样只需在编辑框中删除pick da0a1d4 test这一行。

git fetch

当远程仓库有了新的更新，需要将新的commits更新到本地时，可使用git fetch完成操作。

Usage:

  1. 获取远程主机所有更新。
git fetch <远程主机名>
  1. 获取远程主机特定分支上的所有更新。
git fetch <远程主机名> <分支名>

➜  tests git:(master) git remote -v
trystack    http://git.trystack.cn/openstack/nova.git (fetch)
trystack    http://git.trystack.cn/openstack/nova.git (push)

如上，若想获取将远程trystack仓库nova的newton分支上的所有更新，执行：

git fetch trystack stable/newton

结果如下：

➜  tests git:(master) git branch -a
* master
  remotes/trystack/stable/newton

若想将以上fetch到的远程分支拉取到本地，可基于远程分支切换新的分支，执行：

➜  tests git:(master) git checkout -b stable/newton trystack/stable/newton
Branch stable/newton set up to track remote branch stable/newton from trystack.

同理，若想获取远程仓库master分支的所有更新，执行：

获取远程master分支
# git fetch trystack master  
切换到本地master分支
# git checkout master       
rebase本地master到远程master
# git rebase trystack/master

git tag

  • 创建tag。
git tag -a <tag号> <commit号>
  • 删除多个tag。有如下两种方式。使用正则表达式进行过滤即可。
➜  nova git:(master) git tag | awk '/2012/' | xargs git tag -d
Deleted tag '2012.1' (was 49934aa)
Deleted tag '2012.1.1' (was 29293ce)
Deleted tag '2012.1.2' (was 7a7c8f8)
...

# git show-ref --tag | awk '/2012/ {print ":"$2}' | xargs git push origin
